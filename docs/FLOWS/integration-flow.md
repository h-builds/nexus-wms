# Integration Flow (Phase 4 Backbone)

## Purpose

This document illustrates the full, cross-surface lifecycle of operational data within the NexusWMS ecosystem. It demonstrates how a single physical reality (e.g., an incident) traverses the entirely decoupled architecture—from mobile execution to real-time monitoring and tactical orchestration—using an event-driven backbone.

---

## The End-to-End System Flow 

### 1. Execution (Field-Agent Mobile)
**Location:** Warehouse Floor  
**Action:** An operator identifies crushed packaging. They use the **Field-Agent Mobile** application to register the incident (`POST /api/incidents`).  
**Context:** If offline, the action persists locally. Once online, the sync queue transmits the request.

### 2. Validation & Domain Logic (Backend API)
**Location:** Modular Monolith   
**Action:** The Incidents domain receives the API request, validates properties, and coordinates with the Locations and Product domains. Standardized AI guardrails optionally classify the incident severity.  
**Result:** The incident record is committed to the database. Due to its impact, an internal adjustment movement is orchestrated to block the affected stock. 

### 3. Event Emission (Transactional Outbox)
**Location:** Database & API Server  
**Action:** Before the database transaction commits, the system writes irreversible domain facts (e.g., `incident.reported`, `movement.created`, `inventory.stock.adjusted`) to the `event_outbox` table.  
**Context:** These events are intrinsically linked via identical `correlationId` tracking, guaranteeing a verifiable narrative.

### 4. Broadcasting (Event Bus / WebSocket)
**Location:** Outbox Dispatcher & Laravel Reverb  
**Action:** An outbox dispatcher reads the events and publishes them to the `warehouse.monitoring` WebSocket channel.  
**Context:** The API has finished its job; observation layers independently ingest the real-time facts. Note that the WebSocket channel represents the current MVP transport layer. The Phase 4 architecture is transport-agnostic, and future implementations may scale to message brokers or distributed event buses.

### 5. Operational Observation (Vapor Monitor)
**Location:** Control Room / Supervisor Dashboard  
**Action:** **Vapor Monitor** receives the `incident.reported` and `inventory.stock.adjusted` events over the WebSocket channel.  
**Result:** The Open Incidents KPI increments instantly. The Total Inventory KPI adjusts downwards. The layout visually highlights the affected zone's altered metrics smoothly, without requiring a page refresh.

### 6. Tactical Interpretation (Orchestrator Twin)
**Location:** Tactical Control Station  
**Action:** **Orchestrator Twin** actively monitors the identical event stream.  
**Result:** 
- The 2.5D visual plane instantly spawns structural anomaly markers (e.g., a "crack" or warning pulse) over the explicit `locationId`. 
- The spatial density map is recalculated. 
- A rule-based recommendation is immediately processed given the newly blocked bin.

**Location:** AI Agents (Future)  
**Action:** The **Incident Analysis Agent** registers the event sequence.  
**Result:** The agent evaluates recent frequencies to spot a pattern. It outputs a recommendation (e.g., *"Inspect Packaging Procedures for SKU TV-100"*). The output is stored securely as a **Decision Trace**, linking back structurally to the `causationId` of the event.

## Architectural Transparency

At no point does the **Field-Agent** know about the **Vapor Monitor**. The **API** is oblivious to the **Orchestrator Twin**. The **AI Agent** never alters the database directly. 

Every node communicates purely through immutable facts propagated over the designated Integration Backbone, enforcing absolute loose coupling and modular security.

---

## Correlation Model

Traceability across asynchronous layers and decoupled consumers requires strict identifiers.

- **`correlationId`**: Groups all related events back to a single overarching transaction. It is generated by the API layer upon receiving the initial command request. This ID is passed to every subsequent domain action and event emitted during the entire lifecycle of that request.
- **`causationId`**: Identifies the direct, specific event that triggered the current event. This ensures deterministic reconstruction of event chains (x triggered y, y triggered z), providing absolute historical causality for audits or AI analysis.

**Example Chain:**
1. Action: Mobile incident report.
2. Event 1: `incident.reported` (correlationId: `A`, causationId: `A`)
3. Event 2: `movement.created` (correlationId: `A`, causationId: `Event 1 ID`) 
4. Event 3: `inventory.stock.adjusted` (correlationId: `A`, causationId: `Event 2 ID`)

## Event Lifecycle

To ensure consistent distributed data, every event must strictly follow a canonical lifecycle:

1. **Created**: A domain action (e.g., inventory adjustment) is internally validated and the relevant fact is generated in memory.
2. **Persisted**: The event is written to the `event_outbox` table within the **same database transaction** as the entity mutation. 
3. **Dispatched**: A dedicated background dispatcher or job safely extracts the event from the outbox table.
4. **Broadcast**: The event is pushed through a transport layer (e.g., Laravel Reverb / WebSocket or an event bus).
5. **Consumed**: Frontends, downstream services, and agents intercept the event and react independently.

**Why the Outbox Pattern?**
The transactional outbox guarantees absolute reliability. If the database commit succeeds, the event is guaranteed to exist. If the commit fails, the data is rolled back and no event is erroneously fired. This entirely prevents "lost events" and "ghost mutations".

## Event Types

To maintain modular boundaries and logical safety, NexusWMS strictly delineates between two event classifications:

### Domain Events
- **Definition**: Immutable, historical facts confirming an action that *already occurred*.
- **Source**: Emitted strictly by backend domains (e.g., Inventory, Incidents).
- **Examples**: `incident.reported`, `inventory.stock.adjusted`.

### Derived Signals (Future)
- **Definition**: Advisory conclusions, alerts, or tactical insights derived from analyzing Domain Events.
- **Source**: Emitted by observation layers or AI agents (e.g., shortfalls, bottleneck warnings, layout recommendations).
- **Examples**: `alert.stock.critical`, `recommendation.replenish.generated`. *(Note: These are illustrative examples only. They are NOT part of the current `docs/EVENT_CATALOG.md` and do NOT represent implemented contracts).*

**Why the Separation Matters?**
This distinction prevents systemic side effects. Derived Signals are never blindly trusted; treating them differently actively blocks an AI agent or a predictive algorithm from autonomously mutating core inventory data without formal domain validation. It ensures the operational core remains decoupled from speculative layers.
